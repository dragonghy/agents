## Paper-cuts found

(Each becomes a follow-up Task — listed in priority order.)

1. **No `POST /tickets` REST endpoint.** Ticket creation is only an MCP
   tool (`create_ticket` in `server.py:493`). Validation / smoke / Web
   UI flows that want to create a ticket without an MCP client have to
   shell into SQLite. Add a `POST /api/v1/orchestration/tickets`
   handler in `web/orchestration_api.py`.
2. **MCP `add_comment` timeout cancels in-flight TPM dispatch but the
   SDK subprocess keeps running.** `_with_timeout` wraps `add_comment`
   at 30s (`server.py:286`). The handler awaits `dispatch_comment_to_tpm`
   which awaits `SessionManager.append_message` which awaits the Claude
   SDK turn (typically 60-90s for TPM with tool calls). When the
   wrapper hits 30s, `asyncio.wait_for` raises `CancelledError`, but
   the SDK subprocess (separate process, separate I/O) keeps streaming
   and eventually persists `native_handle` + cost into the session row
   ANYWAY. Net effect: caller sees "timeout / retry" while the TPM
   silently completes, posts comments, and possibly mutates ticket
   state — leaving the system in a half-observed inconsistent state.

   Two-part fix:
   - (a) Comment dispatch should be **fire-and-forget** w.r.t. the LLM
     call. Spawn the `append_message` as a background task; return
     `{"ok": true, "dispatched": true}` immediately.
   - (b) Either lengthen the `add_comment` tool timeout OR (preferably)
     make the dispatch genuinely non-blocking so the timeout becomes
     irrelevant.
3. **No `POST /tickets/{id}/comments` REST endpoint.** Same
   per-channel gap as #1. Web UI / human can't post a comment without
   going through MCP. Add a REST handler that mirrors the MCP tool
   (insert + call `dispatch_comment_to_tpm`).
4. **TPM `mark_ticket_status` bypasses the auto-close hook.** The tool
   in `orchestration_tools.py:286` calls `task_client.update_ticket(...)`
   directly — the SQLite client method, NOT the MCP wrapper that has
   the `maybe_close_tpm_for_status_change` post-hook (`server.py`
   update_ticket handler). So when a TPM marks its own ticket as 0 /
   -1 it does NOT close itself. In this run the TPM was left active
   even though the ticket was archived; manual `POST /sessions/{id}/close`
   was required.

   Fix: have `mark_ticket_status` call the same post-hook (or refactor
   to share a `_apply_ticket_status_change` helper between the MCP
   tool and the orchestration tool).
5. **Legacy v2 dispatcher still spawning tmux windows in the
   background.** Daemon log shows continuous noise:
   ```
   ERROR Failed to spawn session ticket-505-development:
   tmux new-window ... returned non-zero exit status 1
   ```
   every ~14s for tickets 504, 505, 506. Per the "supervisor / watchdog
   removal" note in the progress journal, this was supposed to be
   retired. `dispatcher_v2.py` is still wired into `_start_auto_dispatch_async`
   (`server.py:2133`). Remove the v2 dispatch loop or gate it on a
   feature flag. Drowns the orchestration logs in irrelevant errors.
6. **Profile `mcp_servers` declarative field is stale-warning.** Even
   though TPM tool binding works (via the `mcp_servers` kwarg the
   SessionManager passes), `claude_adapter.py:147-156` still logs
   `MCP wiring is not yet implemented (Task #11). Continuing without
   MCP servers.` every TPM turn. Confusing — the wiring DOES work,
   it's the unrelated `profile.mcp_servers` declarative field that
   isn't honored. Either remove the field from profile.md frontmatter
   or stop logging at INFO when there's no actionable issue.
7. **No way to query "what tools did session X call?" from the API.**
   Had to read JSONL on disk to verify Step 6's tool calls. The
   `GET /sessions/{id}/history` endpoint exists but only renders text;
   tool_use blocks are filtered out. For dogfooding / debugging
   orchestration, a `?include_tool_calls=true` query param would save
   one filesystem trip.
8. **Daemon HTTP `tmux_active` flag is stale.** `GET /api/v1/health`
   returns `{"tmux_active":false}` even though the admin agents tmux
   session is live. Cosmetic but confusing during diagnosis.

## Verdict

**Partial yes.**

The orchestration system **works end-to-end on a real ticket** in the
sense that:

- Status 3→4 PATCH correctly fires `maybe_spawn_tpm_for_status_change`.
- TPM is spawned, registered in the session table, bound to the ticket.
- Comment dispatch correctly looks up the active TPM and forwards the
  message via SessionManager.
- TPM's first LLM turn loads its orchestration tools, reads ticket
  state, posts a status comment, and replies coherently.
- Native session id, cost, and conversation history are persisted in
  the right places (session row + JSONL).
- Manual session close works.

However, **the MCP `add_comment` path is unusable as a wake mechanism
today** because the 30s timeout cancels the wrapper while the
underlying TPM turn (60-90s) silently completes. This means:

- The Telegram bridge (which calls `add_comment` via MCP) cannot
  reliably wake the TPM end-to-end — the wrapper will report timeout
  on every meaningful TPM-driven flow.
- Operators see "timeout / retry" while the system silently keeps
  working, producing a confusing partial-success state.

Once Paper-cut #2 is fixed (fire-and-forget comment dispatch), and
ideally Paper-cuts #1, #3, #4 too, the dogfood loop is genuinely
end-to-end-clean. Until then, validation requires the REST
`/sessions/{id}/messages` path that bypasses the MCP timeout.
