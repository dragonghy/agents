# Executive Log — 2026-04-16

## Critical Architecture Discussion: Project-Level Memory

Human identified a fundamental gap in the system: **agents have ticket-level memory but no project-level memory**. This causes disconnected behavior — each ticket is a standalone task with no awareness of the bigger picture.

### Human's Analysis (verbatim from Telegram)

"Trading 项目的行为不连贯...我们已经到了需要调整 Live Trading 的阶段。这个项目涉及真金白银。Agent 在 trading window 期间每小时 wake up，不只是 monitor，而是要分析数据并决定是否 take action。"

"交易策略只是 Agent 手中的一个工具。我们需要一个稳定的策略和想法，让 Agent 本身来负责记忆和大方向的追踪。Agent 需要去优化交易结果。不能每次 wake up 之后，Agent 就什么都不记得了。"

"之前是以 ticket 为单位进行记忆，但更需要 project level 的记忆。"

### Proposed Hierarchy

```
Project (长期记忆单元)
  └── Milestone (阶段性目标)
       └── Ticket / Task (具体执行单元)
```

都是 ticket 形式，但 level 不同。例如："Live Trading Project" 是一个长期 ticket，所有相关的记忆汇聚在这里。

### Memory Flow (Bottom-Up Aggregation)

1. 每个 task ticket 完成后 → 向上更新所属 milestone/project 的描述
2. Comments 不需要逐条向上 → 每天汇总 (daily aggregate)
3. 结果：project-level ticket 始终有最新的全局 context

### Why This Matters

当 agent 被 spawn 做一个具体 task 时：
1. 读 project ticket → 知道大方向和全局 context
2. 读 milestone ticket → 知道当前阶段目标
3. 读 task ticket → 知道具体要做什么
4. Agent 的行为就连贯了 — 它知道自己在整体中所处的位置

### For Trading Specifically

Human 的要求：
- Agent 不只是执行策略脚本，而是作为**决策者**
- 策略和回测是工具，Agent 做最终决策
- 每小时 wake up → 读 project memory → 分析市场 → 决定 action
- 所有决策记录在 project memory 里，下次 wake up 能续上
