# QA 验收报告审查流程

**本流程是强制性的。** 收到 QA 验收报告后，必须严格按以下步骤逐项检查，不得跳过。

## 第一步：扫描红旗词

在报告中搜索以下关键词。如果核心证据依赖这些内容，直接判定不通过：

- `mock` / `mock测试` — mock 不是 E2E
- `单元测试` / `unit test` — 单元测试是 Dev 的责任，不是 QA 交付物
- `注册成功` / `工具注册` — 工具能注册不代表能用
- `预期行为` / `应该返回` — 这是推测，不是证据

**判断标准**：如果把报告中所有 mock/unit test 的内容去掉，剩下的还能证明功能可用吗？如果不能，打回。

## 第二步：逐功能点检查真实执行证据

对 PRD 中每个验收标准，在报告中寻找以下证据：

### 什么是真实执行证据

| 功能类型 | 合格证据 | 不合格证据 |
|---------|---------|-----------|
| VMware 模块 | 实际 VMware REST API 返回的 JSON（含真实 VM ID、路径） | mock 测试通过、工具注册成功 |
| SSH 模块 | 实际 SSH 到 VM 的命令输出（含真实 IP、真实 stdout） | "支持密码/密钥认证"（描述不是证据） |
| 桌面自动化 | 实际截屏图片、点击后的屏幕变化 | "截屏/点击/输入可用"（结论不是证据） |
| Stub 模块 | Stub 在 VM 内运行 + 聚合器通过 VM IP 调用的 HTTP 日志 | 宿主机本地启动 stub 的测试 |
| 文件操作 | 实际读写文件的内容和路径 | mock 文件系统测试 |
| **UI/前端** | **浏览器截图**（保存为文件，报告中标注路径） | "页面正常显示"、"样式正确"（文字描述不是证据） |

### 关键问题自检

对每个功能点，问自己：
1. **我能在报告中看到真实的输入和输出吗？**（不是描述，是实际数据）
2. **测试环境是否匹配真实使用场景？**（本地 ≠ VM 内，mock API ≠ 真实 API）
3. **如果我是最终用户，这份报告能让我相信功能确实能用吗？**

## 第二步半：前端改动必须验证构建

**涉及前端代码改动时，以下检查是硬性前置条件，不通过直接打回：**

1. QA 报告中必须包含 `npm run build` 的成功输出（不是 `vite dev`，不是 Playwright 通过）
2. `tsc -b` 编译零错误 — **TypeScript 编译失败 = 直接打回，不存在"非阻塞"**
3. 如果报告中没有 build 输出，Product 自己执行 `npm run build` 验证

**原因**：`vite dev` 模式会跳过类型检查，Playwright 测试可以在有 TS 错误的情况下通过。只有 `npm run build`（包含 `tsc -b`）才能验证代码真正可交付。

## 第三步：做出判断

### 通过条件（必须全部满足）

- [ ] 每个 P0 功能点都有真实执行证据
- [ ] P1 功能点至少有基本的真实测试
- [ ] 没有依赖 mock/unit test 作为核心证据
- [ ] 测试环境与真实使用场景匹配
- [ ] **涉及 UI/前端的功能有浏览器截图**（文字描述不算证据，必须有截图文件路径）
- [ ] **前端改动时 `npm run build` 成功**（编译失败不是"非阻塞观察"，是硬性失败）

### 打回时必须说清楚

创建 ticket 给 QA 时，明确写出：
1. **哪些功能点**缺少真实环境证据
2. **你期望看到什么**（例如："我需要看到 ssh_execute 连接到真实 VM 的 stdout 输出，包含真实的 IP 地址和命令结果"）
3. **如果环境有限制**，要求 QA 说明情况，Product 来判断是否可以接受替代方案

## 历史教训

2026-03-01：连续通过了两份不合格报告。
- 阶段一报告：VMware 模块明确标注"mock测试通过"，Product 没有注意到，直接通过
- M4 报告：Stub 仅在宿主机本地测试，未通过 VM IP 验证端到端链路

**根因**：看到"通过"结论就接受了，没有逐项核实证据。不要重蹈覆辙。

2026-03-16：Onboarding (#316) 验收通过了 `tsc` 编译失败的代码。
- `Onboarding.tsx` 缺少 `import { type JSX } from 'react'`，`npm run build` 直接报错
- QA 用 `vite dev` 跑 Playwright 测试（跳过类型检查），报告 "21/21 通过"
- Product 在验收时把编译错误标记为"非阻塞观察"放过了
- 后续 #317 声称 dev-alex 修复了此问题，但修复从未被 commit
- Human 手动测试时发现 build 失败

**根因**：编译失败被当成小问题而不是阻塞性问题。**任何导致 `npm run build` 失败的错误都是 P0，必须打回。**
