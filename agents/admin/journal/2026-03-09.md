## 关键决策
- **彻底禁止 status=2**：这是第三次修复 status=2 滥用问题。之前两次只改了 system prompt 中的约束语句，但 leantime SKILL.md 第 5 节仍然是完整的 status=2 教程（含代码示例），agent 会优先学习 skill 中的具体示例。这次直接删除整个第 5 节，替换为 DEPENDS_ON 模式的教程，从根源上消除了 status=2 的指导。
- **创建 User Agent 角色**：新增 user-sophia，以非技术用户视角测试 Agent-Hub。选择让其在隔离环境中运行完整 agent 团队，而不是在生产环境中测试，避免干扰其他 agent 的工作。
- **Schedule Dispatch 机制**：为 User Agent 设计了定时唤醒功能（每 4 小时），避免 user-sophia 因没有 Leantime 任务而永远不被唤醒。将此机制实现为通用的 schedule dispatch，未来其他 agent 也可使用。
- **Daily Journal 设计**：选择本地 markdown 文件而非 Claude Code memory 系统，因为：文件可被 git 管理、可被其他 agent 搜索引用、格式统一易于解析。日志由 dispatcher 在每天 01:00 触发，按 agent 索引 stagger 5 分钟避免并发。
- **Staleness Detection**（Human 主导实现）：dispatcher 新增 stale ticket 检测——当 agent idle 但有超过 30 分钟的 in_progress ticket 时，发送针对性 prompt 提醒 agent 检查这些 ticket 是否已完成但忘记交付。配置在 agents.yaml 的 `staleness.threshold_minutes`。

## 遇到的问题
- **status=2 反复出现（3 次）**：root cause 是 leantime SKILL.md 中的 status=2 详细教程覆盖了 system prompt 中的禁止声明。教训：当 skill 文档和 system prompt 冲突时，agent 倾向于遵循有具体示例的 skill 文档。
- **Edit tool "no changes" 错误**：尝试编辑已经是目标内容的字符串时触发。解决：重新 read 文件确认实际内容后再编辑。
- **Daemon 重启后 journal 立即触发**：因为 `_journal_dispatched` 是内存状态，重启后为空，导致当天时间已过 01:00 的情况下立即触发所有 agent 的 journal dispatch。这是可接受的行为——agent 会检查当天日志文件是否已存在。

## 经验总结
- **Skill 文档的权重高于 system prompt 的抽象规则**。要禁止某个行为，不能只在 system prompt 写"禁止 X"，必须同时确保所有 skill 文档中没有 X 的具体操作指南。否则 agent 会按照 skill 中的示例操作。
- **Schedule dispatch 的通用化**很有价值。将 user-sophia 的定时唤醒实现为 agents.yaml 中的通用配置，其他 agent 未来也可以配置自己的 schedule。
- **Daily journal 的 dispatcher 集成**比独立 cron job 更好，因为：可以复用 idle 检测逻辑、与其他 dispatch 类型协调优先级、状态跟踪集中管理。
